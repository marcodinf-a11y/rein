# Proposal: Claude Code Hooks for Action Signals

**Target:** AGENTS.md (Claude Code adapter section)
**Priority:** Now
**Effort:** Low (investigation) → Medium (integration)

---

## Proposed Enhancement: Hook-Based Action Logging

The following describes a proposed investigation and potential integration of Claude Code's hook system for pre-execution action signals.

---

### Claude Code Hook Integration

Claude Code provides a hook system that fires callbacks at tool-use boundaries. Two hooks are relevant to the harness:

| Hook | Fires When | Signal Value |
|------|-----------|-------------|
| `PreToolUse` | Before a tool call executes | What the agent is *about to do* — command, file path, arguments |
| `PostToolUse` | After a tool call completes | What the agent *did* — result, exit code, output |

These hooks are the exact interception point that Principal Skinner proposes — and they already exist in Claude Code. The harness does not need to build a custom interception layer.

#### Proposed Use: Action Event Logging

The harness registers hooks that log tool calls as structured events, without blocking execution. This is **observability**, not enforcement — the harness does not deny tool calls.

```json
// .claude/hooks.json (generated by harness per session)
{
  "hooks": {
    "PostToolUse": [
      {
        "type": "command",
        "command": "echo '{\"ts\":\"$(date -u +%Y-%m-%dT%H:%M:%S.%3NZ)\",\"event\":\"tool_use\",\"tool\":\"$CLAUDE_TOOL_NAME\",\"input_preview\":\"$CLAUDE_TOOL_INPUT\"}' >> $HARNESS_EVENT_LOG"
      }
    ]
  }
}
```

The event log captures a timeline of tool calls alongside the existing context pressure stream, enabling the action frequency monitoring proposed in [proposal 04](04_action_frequency_monitoring.md).

#### Why Observability, Not Enforcement

Principal Skinner proposes *blocking* dangerous tool calls. The harness should *log* them instead, for three reasons:

1. **Sandbox containment already prevents damage.** The agent operates in a worktree/tempdir — dangerous commands only affect the disposable sandbox.

2. **Blocking creates an arms race.** Sondera demonstrates this: `rm -rf` blocks are bypassed by `python -c "import shutil; shutil.rmtree('/')"`. Signature-based blocking is inherently incomplete.

3. **Logging provides better signal.** A post-execution report showing "agent ran `rm -rf` 5 times, `curl` to 3 external URLs, and `chmod 777` on 12 files" is more actionable than silently blocking individual calls. The operator can assess the agent's behavior pattern, not just individual actions.

#### Scope Limitation

This integration is Claude Code-specific. Codex CLI and Gemini CLI do not expose equivalent hook systems. For those agents, the harness relies on sandbox containment and post-execution evaluation only.

If the event log format from [ralph-orchestrator proposal 06](../../ralph_orchestrator_deep_dive/proposals/06_event_stream.md) is adopted, hook-generated events would use the same JSONL envelope (`ts`, `event`, `task_id`, `data`), adding a `tool_use` event type.

## Investigation Checklist

Before implementing, the following should be evaluated:

| Question | Why It Matters |
|----------|---------------|
| Can hooks be set per-session without affecting global config? | The harness must not modify the operator's `.claude/hooks.json` permanently. |
| What environment variables are available in hook commands? | Determines what tool-call metadata can be captured (tool name, input, output). |
| Does hook execution add measurable latency? | If PostToolUse hooks add >100ms per tool call, the cost may outweigh the signal. |
| Can hooks write to a file outside the sandbox? | The event log should be in `results/`, not inside the sandbox that gets cleaned up. |
| How do hooks interact with Claude Code's permission model? | If the harness registers hooks, do they conflict with existing permission settings? |

These questions can be answered by reading Claude Code's hook documentation and running a test session with a simple logging hook.

## Rationale

The Principal Skinner deep dive identifies action-content analysis as the harness's primary safety gap: the harness monitors *how much* context is consumed but not *what actions* the agent takes ([synthesis](../00_synthesis.md) §4.4). Principal Skinner proposes a custom interception layer. The critical analysis ([06_critical_analysis.md](../06_critical_analysis.md)) concludes that this is over-engineering for the harness's use case.

Claude Code hooks offer a middle path: gain action visibility without building custom infrastructure. The hooks already exist, are maintained by Anthropic, and provide the exact metadata the harness needs (tool name, input, timing).

This proposal is scoped as "investigation first" because the hook system's interaction with the harness's subprocess model needs validation. If hooks can be configured per-session and write to an external log without measurable latency, integration is straightforward. If not, the harness falls back to sandbox containment + post-execution evaluation, which is already sufficient.
